# Requirements: remove-version-relocate-manifest

**Stage:** Discovery
**Date:** 2026-04-02
**Branch:** feature/remove-version-relocate-manifest

---

## Goal

Remove the harness VERSION file from both the repo and installed targets, relocate the harness manifest JSON from the project root into a `.harness/` subfolder, fix the TMPDIR variable shadowing bug in `install.sh`, clean up stale `.harness-update` sidecar files from previous runs, and ensure every reference to the old paths is updated exhaustively across all scripts, commands, templates, and docs.

---

## Scope

### In scope

- Delete `VERSION` from the repo root
- Delete `src/VERSION` from the distributable template directory
- Update `install.sh` to stop reading, writing, or referencing the `VERSION` file at any point
- Relocate the generated manifest file from `<target>/.harness-manifest.json` to `<target>/.harness/manifest.json`
- Update `install.sh` to write the manifest to the new `.harness/manifest.json` path and read it from the new path during updates
- Update `get_category()` in `install.sh` to remove the `VERSION` entry and add a `harness-owned` entry for `.harness/manifest.json`
- Update the fresh-install flow to read the harness version from the manifest JSON itself (or remove the version field from the generated manifest's source of truth, deriving it another way; see Open Questions)
- Rename the local `TMPDIR` variable in `install.sh` to `HARNESS_TMPDIR` (NB-1 fix) across every occurrence in that file
- Exhaustively update every reference to `VERSION` (as a file path) and `.harness-manifest.json` across the entire repo: all shell scripts, command `.md` files under `.claude/commands/` and `src/.claude/commands/`, documentation files, and any other non-historical file
- Update `.harness/manifest.json` path references in `.claude/commands/seed.md` (the live seed command already uses `.harness/manifest.json` — confirm this is consistent)
- Confirm `src/.claude/commands/seed.md` is brought up to parity with the live `seed.md` on manifest path references
- Remove stale `docs/exec-plans/active/readme-improvements.md` if still present (per cleanup_notes in feature-state.json, this was already removed; verify and document)
- Add `.harness/` to `get_category()` as `harness-owned` so future update runs overwrite it correctly
- Add a `cleanup_stale_sidecars()` function to `install.sh` that removes all `*.harness-update` files under `$TARGET` before per-file processing, in both the update flow and fresh brownfield install flow

### Explicitly out of scope

- Changing the format or fields of the manifest JSON beyond the path relocation
- Migrating NB-3 (missing scaffold file guard) — tracked separately
- Migrating NB-4 (inbox queuing) — tracked separately
- Any changes to agent `.md` files other than command files that reference the old paths
- Changing the external interface of `install.sh` (same flags, same usage pattern)
- Adding automated tests or a test framework
- Any changes to `README.md` or `LICENSE`

## Technical Design

**Author:** Principal Engineer
**Date:** 2026-04-02

### Approach

This feature makes four coordinated changes to `install.sh` and supporting files:

1. **Remove VERSION file** -- Delete the file from two locations and replace all version-derivation logic with a new mechanism that reads `harness_version` from the repo's own generated manifest in the cloned temp directory.
2. **Relocate manifest** -- Move the manifest write target from `<target>/.harness-manifest.json` to `<target>/.harness/manifest.json`, with a migration path that archives the old file during updates.
3. **Reference sweep** -- Update every non-historical file that references the old paths.
4. **TMPDIR rename** -- Mechanical find-and-replace of `TMPDIR` to `HARNESS_TMPDIR` in `install.sh`.

All four changes touch `install.sh` as the primary file. The changes are interdependent (e.g., version derivation depends on manifest relocation), so they should be implemented in a single coordinated pass rather than as independent commits.

---

### Design Decision: Version Derivation (OQ-1)

**Problem:** With `VERSION` file removed, we need a source for the `harness_version` field written into the manifest.

**Chosen approach: Git tag from the cloned repo.**

The installer already clones the repo into a temp directory. The version will be derived by reading the latest Git tag from that clone:

```sh
HARNESS_VERSION="$(cd "$HARNESS_TMPDIR/handoff-harness" && git describe --tags --abbrev=0 2>/dev/null || echo "dev")"
```

**Rationale:**
- Git tags are the idiomatic way to track release versions in a repo without a dedicated version file.
- The cloned repo is already present in both the fresh-install and update flows, so no additional network call is needed.
- The `--depth 1` shallow clone already fetches tags for the checked-out branch.
- The fallback `"dev"` handles the case where no tags exist yet (e.g., pre-release development), ensuring the field is never empty.
- This requires no new files, no build step, and no changes to the repo's release workflow beyond tagging releases (which is already standard practice).

**Version comparison for the update flow's early-exit check:**

The update flow currently compares `REMOTE_VERSION` (from the remote `src/VERSION`) to `LOCAL_VERSION` (from `$TARGET/VERSION`). This will be replaced with:

1. Read the remote version from the Git tag (as above).
2. Read the local version from the existing `$TARGET/.harness/manifest.json` using a new helper function `get_manifest_version()`.
3. Compare the two. If equal, print "Already at version X -- nothing to update." and exit.

The new helper function:

```sh
get_manifest_version() {
  local manifest="$1"
  grep '"harness_version"' "$manifest" \
    | sed 's/.*"harness_version"[[:space:]]*:[[:space:]]*"//; s/".*//'
}
```

This follows the same sed-based JSON extraction pattern already used by `get_manifest_checksum()`.

**Both flows (fresh-install and update) use the same `HARNESS_VERSION` derivation**, keeping the logic unified.

### Design Decision: Manifest Migration (OQ-2)

**Sequence of operations during an update run:**

After cloning and before the per-file processing loop, add a new migration block:

```
Step 1.5 (new): Migrate old manifest if present
  a. If $TARGET/.harness-manifest.json exists:
     - Compute archive name: .harness-manifest.json.YYYY-MM-DD.bak (using the existing $TIMESTAMP variable reformatted as date-only)
     - Rename: mv $TARGET/.harness-manifest.json $TARGET/.harness-manifest.json.YYYY-MM-DD.bak
     - Print: "Migrating manifest: archived .harness-manifest.json as .harness-manifest.json.YYYY-MM-DD.bak"
     - mkdir -p $TARGET/.harness
     - If the old manifest contained a harness_version, use it as LOCAL_VERSION for the early-exit comparison.
  b. If $TARGET/.harness/manifest.json exists:
     - This is the normal case (already migrated). Read LOCAL_VERSION from it.
  c. If neither exists:
     - LOCAL_VERSION="(unknown)" -- triggers the pre-manifest sidecar flow for project-owned files (existing behavior).
```

**Archive naming:** `.harness-manifest.json.YYYY-MM-DD.bak` placed in the target project root (same directory as the original). The date format uses `date +%Y-%m-%d`. If the archive file already exists (e.g., multiple update runs on the same day), append a numeric suffix: `.harness-manifest.json.YYYY-MM-DD.bak.2`, `.bak.3`, etc. However, this edge case is extremely unlikely; the simpler approach of just overwriting the .bak is acceptable since the .bak is itself a backup. **Decision: overwrite the .bak if it already exists.** The user would only run the migration once per install; if they run it twice on the same day, the second run would find `.harness/manifest.json` already present (from the first run's write step) and take path (b), never reaching path (a) again.

**Edge case -- both old and new manifest exist:** If both `$TARGET/.harness-manifest.json` and `$TARGET/.harness/manifest.json` exist, this means a partial migration occurred. The new manifest at `.harness/manifest.json` takes precedence. The old file is still archived for safety, but `LOCAL_VERSION` is read from the new path. The migration block handles this:

```
  a. If $TARGET/.harness-manifest.json exists:
     - Archive it (mv to .bak) regardless
  b. Then, if $TARGET/.harness/manifest.json exists:
     - Read LOCAL_VERSION from it
  c. Else:
     - LOCAL_VERSION="(unknown)"
```

Note: steps (a) and (b) are not mutually exclusive. The archive of the old file always happens if it exists. The version read always prefers the new path.

### Design Decision: get_category for .harness/ (OQ-3)

**Chosen approach: glob pattern `.harness/*`.**

```sh
.harness/*)                           echo "harness-owned" ;;
```

**Rationale:** Using a glob pattern is more maintainable because it automatically covers any future files placed under `.harness/` (e.g., if we later add `.harness/config.json` or `.harness/version`). A specific entry for `.harness/manifest.json` would require updating `get_category()` every time a new file is added to the directory. The glob is also consistent with how `scripts/*`, `hooks/*`, and `.claude/agents/*` are already handled in the function.

### Edge Cases

| Edge Case | Handling |
|-----------|----------|
| No Git tags exist on the cloned repo | `git describe --tags --abbrev=0` fails; fallback `echo "dev"` produces `harness_version: "dev"` |
| Shallow clone does not include tags | `git clone --depth 1 --branch X` fetches tags pointing at the fetched commit. If the tag is on an older commit, it will not be included. Mitigation: add `--tags` to the clone or use `git -c 'versionsort.suffix=-' ls-remote --tags --sort='v:refname'` as a fallback. **However**, for simplicity, the `"dev"` fallback is acceptable. The version field is informational, not load-bearing for installer behavior (the early-exit check will simply not match, causing the update to proceed -- which is the safe default). |
| Old manifest exists but `.harness/` directory also exists (partial migration) | Archive old manifest; read version from `.harness/manifest.json` if it exists |
| Old manifest exists but `.harness/manifest.json` does not (first migration) | Archive old manifest; `LOCAL_VERSION="(unknown)"`; update proceeds fully |
| Neither old nor new manifest exists | `LOCAL_VERSION="(unknown)"`; `MANIFEST_MISSING=true`; all project-owned files get sidecar treatment (pre-manifest behavior, unchanged) |
| `get_manifest_version()` returns empty string | Treated same as `"(unknown)"` -- version comparison will not match, update proceeds. Add a guard: `LOCAL_VERSION="${LOCAL_VERSION:-(unknown)}"` |
| User has customized VERSION file content | Not applicable after deletion -- file no longer distributed or read |
| Stale `.harness-update` files from previous run exist | Cleaned up at start of update or brownfield fresh install, before new sidecars are created |
| No stale `.harness-update` files exist | Cleanup step is silent -- no output |
| Non-sidecar file with `.harness-update` substring in name (e.g., `.harness-update.bak`) | Not matched by the `*.harness-update` pattern -- left untouched |

### Design Decision: Stale `.harness-update` Cleanup (R-7)

**Problem:** Each update run (and brownfield fresh install) may create `.harness-update` sidecar files for project-owned files that were customized by the user. These sidecars accumulate across runs because nothing ever cleans them up. Users must manually find and delete them.

**Chosen approach: Pre-loop cleanup using `find`.**

At the start of each update run (after cloning but before the per-file processing loop), and at the start of each brownfield fresh install (before the per-file copy loop), add a cleanup block that removes all existing `.harness-update` files:

```sh
# Clean up stale .harness-update files from previous runs
STALE_SIDECARS="$(find "$TARGET" -name "*.harness-update" -type f 2>/dev/null)"
if [ -n "$STALE_SIDECARS" ]; then
  STALE_COUNT="$(echo "$STALE_SIDECARS" | wc -l | tr -d ' ')"
  echo "$STALE_SIDECARS" | while IFS= read -r stale_file; do
    rm -f "$stale_file"
  done
  echo "Cleaned up $STALE_COUNT stale .harness-update file(s) from previous run."
fi
```

**Placement in the update flow:** Insert after Step 1.5 (manifest migration) and before Step 2 (read local version). This ensures:
- The cloned repo is available (in case we need it, though cleanup does not depend on it).
- Stale sidecars are removed before the per-file loop at Step 7 creates new ones.
- The cleanup message appears early in the output, before the update changelog.

**Placement in the fresh-install flow:** Insert before the per-file copy loop (before line 280 in the current code). The brownfield detection (`$CONFLICTS`) has already been computed at this point, so we know whether we are in brownfield mode. However, stale sidecars should be cleaned regardless of brownfield status -- a previous brownfield run could have left sidecars even if the current run detects no conflicts.

**Pattern matching:** The `find` command uses `-name "*.harness-update"` which matches only files ending in exactly `.harness-update`. This will NOT match files like `.harness-update.bak` or `.harness-update.2026-04-02`. This is intentional -- only the exact sidecar suffix is targeted.

**Rationale:**
- Cleaning up before the loop (not after) is essential so that the current run's new sidecars survive.
- Using `find` rather than a glob ensures subdirectories are covered (sidecars can exist at any depth under `$TARGET`, e.g., `docs/CONTRIBUTING.md.harness-update`, `.claude/commands/seed.md.harness-update`).
- The count message provides visibility without being noisy. Silent when there is nothing to clean.

**Edge case -- user intentionally kept a `.harness-update` file:** This cleanup is opinionated: it assumes all `.harness-update` files are stale. If a user was in the middle of reviewing sidecars from a previous run and then runs another update, the old sidecars will be removed. This is acceptable because (a) the new run will recreate sidecars for any files that still have conflicts, and (b) leaving stale sidecars is worse than removing them, since the stale content is outdated anyway.

**Shared helper approach:** Since the same cleanup logic is needed in both the update flow and the fresh-install flow, extract it as a function to avoid duplication:

```sh
cleanup_stale_sidecars() {
  local target="$1"
  local stale_sidecars
  stale_sidecars="$(find "$target" -name "*.harness-update" -type f 2>/dev/null)"
  if [ -n "$stale_sidecars" ]; then
    local stale_count
    stale_count="$(echo "$stale_sidecars" | wc -l | tr -d ' ')"
    echo "$stale_sidecars" | while IFS= read -r stale_file; do
      rm -f "$stale_file"
    done
    echo "Cleaned up $stale_count stale .harness-update file(s) from previous run."
  fi
}
```

Place this function alongside the other helper functions (`compute_sha256`, `get_manifest_checksum`, `get_manifest_version`) near the top of the script. Call it in both flows:
- Update flow: `cleanup_stale_sidecars "$TARGET"` after Step 1.5
- Fresh-install flow: `cleanup_stale_sidecars "$TARGET"` before the per-file copy loop
